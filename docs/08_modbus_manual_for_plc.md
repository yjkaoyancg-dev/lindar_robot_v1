## 1. 文档目的

本文档用于指导下位机开发人员完成以下工作：

1. 通过 Modbus RTU 向 `robot_plc_crontorl` 从站发送启动命令。
2. 轮询状态寄存器，判断当前是否识别完成。
3. 在完成状态下读取位置、姿态和置信度寄存器。
4. 正确完成一轮“启动 -> 等待 -> 读取 -> 复位 -> 下一轮启动”的流程。

本文档同时包含：

- 寄存器映射

- 32-bit 浮点寄存器解码方法
- 调试命令与示例
- 已验证的联调结论
- 常见问题与排查方法

## 2. 系统整体说明

当前系统由以下模块组成：

1. **雷达采集模块** `lidar_node`
2. **目标检测模块** `range_density_detector_node`
3. **PLC 桥接模块** `range_plc_bridge_node`
4. **Modbus RTU 从站模块** `robot_plc_crontorl_node`
5. **PLC 主站 / 假主站工具**
   - 实际下位机 / PLC 主站
   - 或联调用 `mbpoll`

### 2.1 功能链路

下位机视角下，完整业务流程为：

1. 我方服务端启动并等待。
2. PLC 主站向命令寄存器写入启动命令。
3. 从站切换到“正在启动”状态，清零数据区。
4. 从站调用内部触发服务，等待本轮新识别结果。
5. 一旦收到本轮合法结果：
   - 将位置/姿态/置信度写入寄存器
   - 状态切换为“启动完成”
6. PLC 主站读取完成状态及结果寄存器。
7. PLC 主站写回 `cmd=0` 复位，准备下一轮。

## 3. Modbus 通信参数

### 3.1 当前联调参数

- 协议：**Modbus RTU**
- 从站地址（Slave ID）：**1**

- 校验：**N**（无校验）

## 4. 状态机说明

系统只使用 **3 态状态寄存器**：

- `0`：未启动（IDLE）
- `1`：正在启动 / 等待结果（WAITING）
- `2`：启动完成 / 结果有效（READY）

### 4.1 状态含义

#### 状态 0：未启动

- 系统处于空闲/复位状态
- 数据寄存器全部为 0
- 可以接受下一轮启动

#### 状态 1：正在启动

- 已检测到命令寄存器从 `0 -> 1` 的上升沿
- 本轮旧数据已经失效
- 位置/姿态/置信度寄存器必须全部为 0
- 系统正在等待本轮新识别结果

#### 状态 2：启动完成

- 本轮识别结果已经写入寄存器
- 数据寄存器有效
- PLC 主站可以读取位置、姿态和置信度

### 4.2 重要规则

1. **只有 `state=2` 时，数据寄存器才可视为有效。**
2. `state=0` 或 `state=1` 时，数据寄存器必须认为无效。
3. **新一轮启动前，主站必须先把 `cmd` 写回 0。**
4. 再次启动时，必须重新形成 `0 -> 1` 上升沿。
5. 不能在 `cmd` 一直保持 1 的情况下重复触发新一轮。

## 5. 寄存器映射

### 5.1 代码内部 0-based 映射

系统内部定义：

- `cmd`：地址 `0`
- `state`：地址 `1`
- `position`：起始地址 `2`
- `attitude`：起始地址 `8`
- `confidence`：起始地址 `14`

### 5.2 对 PLC / mbpoll 看到的 1-based 读写地址

大多数主站工具（如 `mbpoll`）按 **1-based** 显示寄存器，因此实际看到的是：

| 主站看到的寄存器号 | 含义 |
|---|---|
| `[1]` | `cmd` |
| `[2]` | `state` |
| `[3] [4]` | `x`（float） |
| `[5] [6]` | `y`（float） |
| `[7] [8]` | `z`（float） |
| `[9] [10]` | `roll`（float） |
| `[11] [12]` | `pitch`（float） |
| `[13] [14]` | `yaw`（float） |
| `[15] [16]` | `confidence`（float） |

### 5.3 含义总结

- `cmd`：PLC 主站写
  - `0`：空闲/复位
  - `1`：请求启动

- `state`：从站写
  - `0`：未启动
  - `1`：正在启动
  - `2`：启动完成

- `x,y,z,roll,pitch,yaw,confidence`：从站写，PLC 主站读

## 6. PLC 主站标准流程

这是下位机应采用的推荐流程。

### 第 1 步：系统启动

在 Ubuntu 侧执行：

```bash
~/stop_photonix_stack.sh
~/start_photonix_stack.sh
```

等待我方服务全部启动完成。

### 第 2 步：发起一次新轮次前先复位

PLC 主站先写：

- `cmd = 0`

目的是：

- 清掉上一轮命令状态
- 为新的 `0 -> 1` 上升沿做准备

### 第 3 步：发送启动命令

PLC 主站写：

- `cmd = 1`

这会触发一次新轮次启动。

### 第 4 步：轮询 `state`

主站持续轮询状态寄存器：

- 若 `state = 1`：表示系统仍在处理，继续等待
- 若 `state = 2`：表示本轮结果已经准备好，可以读取数据寄存器

### 第 5 步：读取位姿与置信度

当 `state = 2` 时，读取：

- `[3]-[16]`

并按大端 float 解码：

- `x = [3],[4]`
- `y = [5],[6]`
- `z = [7],[8]`
- `roll = [9],[10]`
- `pitch = [11],[12]`
- `yaw = [13],[14]`
- `confidence = [15],[16]`

### 第 6 步：本轮结束后复位

读取完成后，主站应写：

- `cmd = 0`

这样系统回到可触发下一轮的状态。

## 7. 轮询策略建议

推荐 PLC 主站使用如下策略：

1. `cmd=0`
2. 延时 `100~1000 ms`
3. `cmd=1`
4. 开始轮询 `state`
5. 每 `100~500 ms` 读一次 `state`
6. 若 `state=2`，立即读取结果寄存器
7. 结果读取完成后写回 `cmd=0`

### 7.1 强烈建议

**不要在 `state=1` 时提前读取位置/姿态并当成有效值。**

因为按协议设计，`state=1` 时数据区本来就应该是全 0。

## 8. 32-bit 浮点解码方式

系统采用：

- 每个 float 占 2 个 holding register
- 按 **big-endian** 解码

也就是：

- 第一个寄存器是高 16 位
- 第二个寄存器是低 16 位

### 8.1 Python 解码示例

```python
import struct

# hi, lo 为两个 16-bit holding register

def decode_f32_be(hi, lo):
    raw = struct.pack('>HH', hi & 0xFFFF, lo & 0xFFFF)
    return struct.unpack('>f', raw)[0]

x = decode_f32_be(reg3, reg4)
y = decode_f32_be(reg5, reg6)
z = decode_f32_be(reg7, reg8)
```

### 8.2 为什么单个寄存器会显示负数

有时主站工具会把单个 16-bit 寄存器同时显示为无符号/有符号值，例如：

- `48476 (-17060)`

这是正常现象。因为：

- 单个寄存器本来只是一个 32-bit float 的一半
- 不能把单个寄存器单独解释成完整数值

必须成对组合后再按 float 解码。

## 11. 常见问题

### 问题 1：`state=1`，数据全 0

这是正常现象。表示：

- 系统已经收到启动命令
- 但本轮识别结果尚未写入

PLC 主站应继续轮询，不要把这时的数据当成有效值。

### 问题 2：`state=2`，但数据全 0

这不是期望状态。出现这种情况通常说明：

- 桥接层错误发布了零值结果
- 或识别结果时序有问题

当前联调已经避免这种“假完成态”继续被使用。

### 问题 3：一直停在 `state=1`

通常表示：

- 本轮结果还没有准备好
- 检测链路没有在规定时间内产出新结果
- 或桥接层还在等待有效新数据

### 问题 4：Modbus 超时

优先检查：

1. 主从角色是否分配正确
2. 两只 USB485 是否用对口
3. A/B 线是否正确
4. 是否有多个从站进程同时占用同一个串口
5. 波特率、校验、从站地址是否一致

## 12. 对下位机开发人员的最终约束

下位机开发时，请严格遵守以下规则：

1. **每一轮开始前先写 `cmd=0`。**
2. **再写 `cmd=1` 形成上升沿。**
3. **只在 `state=2` 时读取数据并当成有效结果。**
4. **读取完成后再次写 `cmd=0`，准备下一轮。**
5. **不要在 `state=1` 时使用数据寄存器内容。**
6. 若连续多轮工作，必须每轮都执行完整的 `0 -> 1 -> 轮询 -> 读取 -> 0` 流程。
